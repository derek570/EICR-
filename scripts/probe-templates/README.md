# scripts/probe-templates/

Templates the optimizer's signature-detection layer uses to write
auto-generated regression probes. Created by Cluster 3 Item 8 of the
optimizer rewrite (see `.planning/optimizer-rewrite-plan-2026-06-03-final.md`).

## What lives here

- `garbles/probe_*.yaml` — Flux / dialogue-engine garble templates.
- `dispatcher_gaps/probe_*.yaml` — off-enum / out-of-range value templates.
- Other `<suite>/` subdirectories will appear as new bug-class signatures
  are added to the `KNOWN_BUG_SIGNATURES` registry in
  `scripts/analyze-session.js`.

## Naming convention

- File path matches the registry entry's `probe_template` field exactly:
  `scripts/probe-templates/<suite>/probe_<bug-class-id>.yaml`
- The `<suite>` segment also determines where the writer drops the
  generated YAML: `tests/fixtures/voice-latency-scenarios/auto-generated/<suite>/`.

## Placeholder syntax

Templates use `scripts/render-prompt.cjs`'s flat `{{KEY}}` substitution
so we don't have a second templating tool to maintain. Common placeholders:

| placeholder | source |
|---|---|
| `{{SESSION_ID}}` | UUID of the source session that matched the signature |
| `{{TIMESTAMP_ISO}}` | when the probe was generated (writer wall-clock) |
| `{{CIRCUIT_REF}}` | circuit number from the signature's hit data |
| `{{TRANSCRIPT}}` | verbatim user utterance that produced the match |
| `{{EXPECTED_VALUE}}` | what the field SHOULD contain (post-fix) |
| `{{ACTUAL_VALUE}}` | what the field DID contain (pre-fix) |

Signature-specific placeholders are documented at the top of each
template file.

## Output location

The writer (Cluster 3 Item 8 — wired into `session-optimizer.sh` after
the recommendation-generation step) writes populated probes to:

    tests/fixtures/voice-latency-scenarios/auto-generated/<suite>/probe_<id>_<timestamp>.yaml

This is the "auto-generated" sibling directory locked in Decision 2 of
the optimizer rewrite plan. The default harness sweep and CI do NOT
pick it up — an explicit walker skip in
`scripts/voice-latency-bench/transcript-replay.mjs` (`if (!INCLUDE_AUTO_GENERATED && entry.name === 'auto-generated')`)
is the enforcement point.

## Promotion to the main regression suite

Auto-generated probes are CANDIDATES, not regression guards. Promotion
is manual:

1. Verify the probe actually reproduces the bug by running
   `node scripts/voice-latency-bench/transcript-replay.mjs --scenario=<auto-generated path> --include-auto-generated`
   against staging. Expect FAIL (or FAIL → PASS after the proposed fix lands).
2. If verified, `mv` the file from `auto-generated/<suite>/` to the
   main `<suite>/` directory. The harness then picks it up on every
   sweep.
3. If the probe doesn't reproduce, leave it under `auto-generated/` for
   audit (or delete it).

## Why a sibling directory, not the main one with a flag

Option A (probes in the main directory with `auto_generated: true`) was
rejected during plan refinement. The failure mode: a falsely-detected
bug class auto-adds a probe that then silently masks a real regression
in CI — the kind of silent failure that's painful to debug later.
Option B keeps the main probe suite as the manually-curated trustworthy
regression bed.

## Writer implementation (deferred to follow-up)

The writer hook lives in `session-optimizer.sh` after the
recommendation generation step. Pseudocode:

```sh
# For each bug_signature_hits entry with a probe_template:
for hit in $(jq -c '.bug_signature_hits[]' analysis.json); do
  TEMPLATE_PATH=$(echo "$hit" | jq -r '.probe_template // empty')
  [ -z "$TEMPLATE_PATH" ] && continue
  [ ! -f "$CODEBASE/$TEMPLATE_PATH" ] && continue
  # Build a flat vars JSON from the hit's session-level context.
  VARS_FILE=$(mktemp)
  jq -n --arg sid "$SESSION_ID" --arg ts "$(date -u +%FT%TZ)" ... > "$VARS_FILE"
  SUITE=$(basename "$(dirname "$TEMPLATE_PATH")")
  OUT_DIR="$CODEBASE/tests/fixtures/voice-latency-scenarios/auto-generated/$SUITE"
  mkdir -p "$OUT_DIR"
  OUT_FILE="$OUT_DIR/probe_$(jq -r '.id' <<< "$hit")_$(date +%s).yaml"
  node "$SCRIPTS_DIR/render-prompt.cjs" "$CODEBASE/$TEMPLATE_PATH" "$VARS_FILE" "$OUT_FILE"
  # Inject the generated_probe_path back into the recommendation
  # corresponding to this signature so the HTML renderer can link it.
  rm -f "$VARS_FILE"
done
```

The full implementation needs to populate signature-specific placeholders
(e.g. `LL_VALUE` + `LE_VALUE` for ir_bare_bridge), which depends on each
hit carrying the raw values_snapshot it matched against. Today's hit
shape includes `recommend_shape.metadata`; extending it to also carry
a verbatim copy of the matched event's data would close that gap.
