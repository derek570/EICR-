## Why this approach

### Per-file MSW, not a global setup

`tests/setup.ts` is shared by the 52 prior tests — several of them (`api-client.test.ts`) stub `globalThis.fetch` directly. Installing MSW globally would either fight those stubs or force a test-by-test audit to thread handlers through. A per-file factory lets integration tests opt in; regression tests stay untouched.

### `mountHook` instead of `@testing-library/react`'s `renderHook`

This was the load-bearing design call of Wave 3a and also the blocker that cost the most time. Documented in full here so the next agent doesn't re-discover it.

**The problem.** The monorepo root hoists React 19.2.3 (pulled in by `@dnd-kit/utilities`'s range); `web` declares 19.2.4. Product code + Vite-transformed test files go through Vite's resolver, so they get web's copy. But `@testing-library/react` lives at the root `node_modules` and its CJS dist calls `require('react')` — Node's bare-require resolution walks up from the dist location, finds the root's `react`, and stops there. Result: two React instances coexist in the same test process; the hook-under-test uses instance A, RTL's renderer uses instance B, the dispatcher is installed on B but read from A, and every `useRef` throws "Invalid hook call".

**Fixes attempted that didn't stick:**
1. `resolve.alias` — applies only to imports going through Vite's resolver. RTL's CJS `require` bypasses it.
2. `server.deps.inline: ['@testing-library/react', ...]` — inlining does force Vite transform on CJS, but vitest 4's dep-optimiser still resolves nested `require('react')` via Node and keeps the root copy.
3. `ssr.noExternal: [...]` — same root cause as (2).

**Fix that worked.** Sidestep RTL entirely. `mountHook` (defined in `outbox-replay.integration.test.tsx`) mounts a tiny `<Host>` component with `createRoot` from `react-dom/client`, runs the hook inside, and exposes an `unmount()` handle. Because the test file is Vite-transformed, `react` + `react-dom/client` resolve to the aliased web-local copies, and the whole render flow runs against a single React instance. The alias in `vitest.config.ts` is kept as belt-and-braces for any future product file that does something exotic with React imports.

The `mountHook` helper is ~20 lines, comment-for-comment documented, and intentionally inline rather than extracted — the next agent should be able to read the test file top-to-bottom without jumping into a helper module to understand how the hook is actually mounted.

### No `vi.useFakeTimers()` in the 5xx case

The spec says `vi.useFakeTimers()` for the 5xx backoff test. In practice that hangs the run — `waitForOutbox` polls IDB via `setTimeout`, and faking the timer base starves the poll loop. Fake timers also require a separate reset (`vi.useRealTimers()` in `afterEach`), and the initial install produced an extra "The current testing environment is not configured to support act(...)" warning that muddied the output.

The replacement contract (documented inline in the test): rely on `outbox.ts`'s `BASE_BACKOFF_MS = 2_000` and `scheduleNext`'s 1s floor. The first failed attempt sets `nextAttemptAt` at least 1s in the future, which is well past our ≤100ms assertion window. The test asserts `hitCount === 1` at the end to prove no second pass slipped through before the unmount torn down the reschedule timer. Same safety guarantee as fake timers, without the flake surface.

### Adapter-typed fixtures

Every MSW response body is built by calling `SaveJobResponseSchema.parse({ success: true })` or `JobDetailSchema.parse(...)` at module scope. If the schemas evolve in a way the fixture no longer satisfies, the test file fails at import time — not silently several minutes later when the fetch response is consumed. This is the D7 observability contract ("keep test fixtures in lockstep with the wire shape") paid in the simplest possible way.

### No product-code changes

The task's hard constraint was "do not modify product code unless a test uncovers a regression." Kept. No changes to `outbox.ts`, `outbox-replay.ts`, `queue-save-job.ts`, or `job-cache.ts`.

## Remaining gaps (genuinely deferred)

### Non-asserted: row 1's state in the sign-out case

Test 4 ("sign-out mid-flight") only asserts that **row 2** is untouched after unmount. Row 1's fate depends on whether the in-flight `api.saveJob` promise had time to call `removeMutation` + `putCachedJob` before the `cancelledRef.current = true` cleanup fired — both outcomes are acceptable under the current semantics. A tightened variant of this test would need either:

- A product-code change exposing a synchronous `cancelReplay()` that interrupts the current `attempt()` (the task description's wording, but no such export exists), or
- A mock on `removeMutation` / `putCachedJob` to assert the post-settle cleanup is skipped.

Neither is in Wave 3a's scope. Flagged here so the next wave considering a `cancelReplay` export knows the test is ready to tighten.

### No product-code bugs uncovered

The five cases ran clean on the first pass once the React-instance pin was in place. The only issues surfaced were harness issues (RTL's React resolution, fake-timers incompatibility with the poll helper) — both solved in this wave. No `outbox-replay.ts` or `outbox.ts` behaviour was wrong.

### Wave 3b candidates (explicitly out of scope here)

Per FIX_PLAN D6 tier-2 + E2, other integration surfaces that would benefit from the same MSW harness but aren't replay-path:

- `JobProvider.updateJob` optimistic UI + server merge race
- Dashboard SWR read-through cache vs network race (IDB cache warm + network 304 path)
- Login redirect rules (`auth-redirect.ts` already has unit coverage; the cookie-vs-bearer + 401 refresh round-trip is an MSW natural fit)

All three can reuse `tests/msw-server.ts` unchanged — the factory is generic.

### `onUnhandledRequest: 'error'` may surface service-worker noise in future

If a future test renders a component that registers a Serwist `navigator.serviceWorker.register()` side effect, MSW will log an unhandled-request error for the SW asset fetches. Workaround is per-test: add a pass-through `http.get(...)` for the SW assets. Not an issue today; flagged here so the next agent knows why they might see an "unhandled request" stderr on an unrelated test.
